# Anti-Patterns

## Purpose

Document the planning and product behaviors that would make `wegotthis.online` heavier, noisier, or harder to trust.

## Current Decisions

- The project should name anti-patterns early so the team can avoid normalizing them.
- Calm software principles should apply to product behavior and to project management behavior.
- GitHub and Notion should have distinct roles instead of competing as duplicate planning systems.

## Anti-Patterns

## Everything Becomes a Task

- Not every idea, note, or concern needs to become a tracked task immediately.
- Converting every thought into task overhead creates noise and weakens prioritization.

## Dashboard Overload

- Dashboards should not become a collection of decorative widgets, charts, and low-value metrics.
- If a dashboard makes people think harder instead of orienting faster, it is failing.

## Workflow Rigidity

- Workflows should guide people without forcing every church into a brittle operating model.
- Overly rigid flows create resistance, bypass behavior, and administrative resentment.

## Notification Spam

- Sending too many alerts destroys trust in the system.
- Notifications should support action, not simulate activity.

## Enterprise Process Creep

- The project should resist turning into a heavy enterprise platform with endless approvals, policy layers, and administration burden.
- If the system requires too much management to remain useful, it is missing the point.

## Over-Normalized Notion Systems

- Notion should support active execution, not become a complex internal product of its own.
- Too many databases, status layers, or mirrored structures create coordination drag.

## Duplicated Source of Truth Between GitHub and Notion

- Durable architecture and decisions should not be maintained in parallel with conflicting versions.
- GitHub holds canonical documentation. Notion manages active execution.

## Feature Gravity

- New ideas should not automatically pull themselves into MVP.
- Features should enter through phase planning, category review, and readiness checks.

## Excessive Configuration

- The platform should not solve uncertainty by adding endless settings, toggles, builders, and rules.
- Strong defaults are usually better than broad configuration surfaces.

## Building Integrations Before Core Workflows Stabilize

- Optional connectors should not be built before the native product flow is useful on its own.
- Integration work should follow proven workflow stability, not wishful sequencing.

## Open Questions

- Which anti-patterns should be reviewed explicitly during future design and implementation work?
- Which anti-patterns need concrete review checklists once the app exists?

## Notes

- Use this document as a constraint when evaluating roadmap additions, UI decisions, workflow design, and execution process changes.
- If a proposed improvement increases confusion, duplication, or administrative drag, it likely belongs here.
