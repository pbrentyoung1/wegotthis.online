# Approval Flow Map

## Purpose

Describe how review, approval, rejection, and requested changes should move through the system.

## Core Rules

- Keep approval flows centered on deliverables in MVP.
- Support internal and external review without requiring full external accounts.
- Preserve approval notes and requested changes in context.
- Keep approval logic calm and understandable.

## Relationships

- Approvals attach primarily to deliverables.
- Approval actions may be taken by internal users or scoped magic-link reviewers.
- Approval outcomes should influence deliverable state, project history, and notifications.
- Requested changes should return work to revision without losing history.

## Open Questions

- Who can override, reopen, or waive approvals?
- Which approval actions should vendor-scoped External Reviewers be allowed to take?
