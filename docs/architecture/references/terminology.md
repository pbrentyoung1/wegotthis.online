# Terminology

## Purpose

Provide a shared vocabulary for product, workflow, security, and data diagrams.

## Core Rules

- Use the same terms across roadmap, product docs, and diagrams.
- Prefer product language over technical shorthand where possible.
- Record term changes here when naming decisions shift.
- Avoid inventing synonyms for established objects.

## Relationships

- Terminology should align with product structure, workflow maps, permission planning, and data diagrams.
- Naming consistency should support GitHub documentation and Notion execution views.
- Shared language should reduce confusion across design, planning, and implementation.

## Open Questions

- Which terms still need stronger definitions before scaffolding?
- Which objects are most likely to be mislabeled across product and technical docs?
- Which stakeholder-facing terms should differ from internal technical terms?
