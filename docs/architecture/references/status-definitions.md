# Status Definitions

## Purpose

Provide a durable reference for status names, meanings, and boundaries across the system.

## Core Rules

- Status labels should be simple, stable, and human-readable.
- Status names should not duplicate unrelated concepts.
- Status decisions should be documented before serious schema work.
- Avoid status sprawl that creates confusion instead of clarity.

## Relationships

- Status definitions support lifecycle maps, permission planning, notifications, and reporting.
- Status language should connect to state transitions and workflow visibility rules.
- Shared status definitions should help keep product, design, and data planning aligned.

## Open Questions

- Which statuses are global versus object-specific?
- Which statuses must exist in MVP?
- Which status changes should trigger notifications, approvals, or audit events?
