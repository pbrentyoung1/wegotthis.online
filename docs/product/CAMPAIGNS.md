# Campaigns

## Purpose

Define campaigns as optional parent containers above projects.

## Current Decisions

- Working decision: Campaigns -> Projects -> Deliverables -> Tasks.
- Campaigns are optional.
- Simple work can start directly as a project.
- Projects may belong to a campaign but do not have to.

## Use Cases

- Easter
- Christmas
- Sermon arcs
- Capital campaigns
- Major church-wide initiatives
- Multi-department promotion efforts

## Campaign Value

- Visibility across related projects.
- Coordination across teams and departments.
- Reporting by large initiative.
- Asset grouping across many projects.
- Shared context for timelines and messaging.

## Open Questions

- Are campaigns MVP or early post-MVP?
- What campaign fields are required?
- Should campaigns have high-level planning tasks later?

## Notes

- Campaigns should help organize large initiatives without forcing every project into a parent container.
