# UI Principles

## Purpose

Define product interface principles for design and implementation.

## Current Decisions

- Inspinia is expected to provide the UI/admin foundation.
- The application should prioritize maintainability and clarity.
- The UI should feel breathable, steady, and clear.
- Whitespace should be treated as emotional space that helps people think.
- The interface should feel like a thoughtfully designed creative workspace: ordered, warm, bright, and ready to help.
- The interface should not feel like a compliance engine, HR portal, IT dashboard, or clunky church database.
- Motion should feel subtle, responsive, and calm, as if the interface is listening.
- Notifications, errors, and empty states should feel helpful and human rather than alarm-heavy.

## Open Questions

- How closely should the product follow Inspinia defaults?
- What components should be standardized first?
- How dense should operational screens be?
- How can conversational guidance be introduced without slowing down expert users?

## Notes

- UI principles should be tested against real church communications workflows.
- Avoid dense, harsh, alarm-heavy dashboards.
- Avoid aggressive red or error-heavy states except when truly necessary.
- Power users still need speed, shortcuts, batch actions, and efficient workflows.
