# File Storage

## Purpose

Document how uploaded files, media assets, references, and generated outputs may be stored.

## Current Decisions

- Media asset management is a core product area.
- Media assets should support metadata and project relationships.
- Working direction: use storage abstraction so physical storage can change without rewriting core product logic.
- Working direction: separate physical file storage from searchable metadata.
- Working direction: project closeout should use a consistent archive folder structure.
- Working decision: abstract file storage from day one.
- Local server storage will be used for the initial single-tenant proof/staging build.
- Production files may live on the application server during early single-tenant proof/staging, as long as access is controlled through Laravel storage and file metadata remains in PostgreSQL.
- The current hosting.com/cPanel environment is not the final serious production storage decision.
- S3-compatible object storage should be introduced when the product moves to multi-tenant hosting.
- Cloudflare R2 or another S3-compatible provider can be evaluated at the multi-tenant transition.
- Thumbnails and previews should be generated through queued jobs.
- CDN delivery should be considered for performance once production file storage is selected.
- Future Google Docs, Dropbox, and OneDrive connectors should be treated as external references or sync integrations, not as the primary storage model for the initial build.

## Early Strategy Notes

- Storage abstraction should support local development, initial local server proof/staging storage, hosting.com/cPanel constraints, and future object storage.
- Local server storage is acceptable for early single-tenant proof/staging if backups, permissions, and disk limits are documented.
- Future S3 or Cloudflare R2 compatibility should be considered before hard-coding storage assumptions.
- File paths should be tenant-aware and migration-friendly even before true multi-tenancy ships.
- Final deliverables, source files, exports, vendor files, photos, video, copy, approvals, and notes may need different retention and access rules.
- Media Library metadata should store searchable context such as project, department, status, usage, version, uploader, people in image, and reusable tags.
- Physical file paths should not be the only source of truth for finding assets.
- Archive folder structure should align with `/docs/product/PROJECT_CLOSEOUT.md`.

## Phased Storage Model

### Phase 1: Initial Single-Tenant Build

- Use Laravel local disk for development and initial production.
- Store uploads outside public web access unless explicitly published through controlled routes or temporary links.
- Use Laravel storage links only for assets that are safe to expose publicly.
- Keep database records as the source of truth for file owner, tenant/project/deliverable context, visibility, version, rights metadata, and retention state.
- Back up uploaded files alongside database backups so metadata and physical files remain aligned.

### Phase 2: Multi-Tenant Transition

- Move production file storage to S3-compatible object storage.
- Preserve the same application storage abstraction so file access rules do not change.
- Migrate local server files into tenant-scoped object storage paths.
- Use signed temporary URLs for private file access and magic-link review flows.
- Consider CDN delivery for approved public assets, previews, and generated thumbnails.

## Notes

- File storage decisions must account for hosting.com, cPanel, backups, disk quotas, and future scaling.
- Do not create storage implementation code until application scaffolding is explicitly requested.
- File storage should support project closeout and Media Library search without assuming every file lives in the same place forever.
- Archive packages should be planned as part of project closeout.
- Searchable metadata should live in the database even when files live in local storage, R2, S3-compatible storage, CDN-backed storage, or future providers.
- External document/file connector metadata should also live in the database so project context remains searchable even when the source file lives in Google Docs, Dropbox, OneDrive, or another service.
